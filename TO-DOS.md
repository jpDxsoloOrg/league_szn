## Database Interface Layer (Repository Pattern) - 2026-04-17 14:01

- **Plan and implement a database abstraction layer so the data source is swappable** - Introduce a repository/DAO interface layer between Lambda handlers and the underlying database, so replacing DynamoDB with Postgres/Neon/SQLite/etc. only requires writing a new implementation of the interface — no handler changes. **Problem:** Every Lambda handler currently imports `dynamoDb` directly from `backend/lib/dynamodb.ts` and calls DynamoDB-specific operations (`get`, `put`, `update`, `scan`, `scanAll`, `query`, `transactWrite`). DynamoDB key shapes (PK/SK, GSI names like `TournamentIndex`, `PlayerIndex`), table names from env vars, and even ExpressionAttributeValues leak into business logic across ~80+ handlers. This makes it impossible to swap the data source — the handlers ARE coupled to DynamoDB. Note `backend/neon/` already exists as untracked exploration, signaling real intent to evaluate Postgres. **Files:** `backend/lib/dynamodb.ts` (current low-level wrapper — the leaky abstraction), `backend/functions/**/*.ts` (~80+ handlers importing `dynamoDb` directly — grep for `from '../../lib/dynamodb'` to enumerate), `backend/serverless.yml:TABLE_NAME env vars` (table names currently injected per-function), `backend/neon/` (untracked — existing Postgres exploration), data model domains from CLAUDE.md: Players, Matches, Championships, ChampionshipHistory, Tournaments, Seasons, SeasonStandings, Divisions, Events, ContenderRankings, RankingHistory, FantasyConfig, WrestlerCosts, FantasyPicks, SiteConfig, Promos, Challenges, Stipulations, MatchTypes. **Solution:** Design a plan (no code yet — just the plan file under `docs/plans/plan-database-interface-layer.md`) that covers: (1) One repository interface per domain aggregate (e.g., `PlayersRepository`, `MatchesRepository`, `StandingsRepository`) defined in `backend/lib/repositories/*.ts` — methods expressed in domain terms (`findById`, `listBySeason`, `recordMatchResult`) NOT DB terms (no `pk`/`sk`/`GSI` in signatures). (2) A `DynamoRepositories` implementation wrapping current `dynamoDb` calls. (3) A factory/DI container (`backend/lib/repositories/index.ts`) that selects the implementation via env var (e.g., `DB_DRIVER=dynamo|postgres`). (4) Transaction boundary strategy — the hardest part, since `recordResult.ts` does cross-table transactional writes; the interface needs a `UnitOfWork` or `runInTransaction(fn)` abstraction that both DynamoDB `TransactWriteItems` and Postgres `BEGIN/COMMIT` can satisfy. (5) Migration order — start with a read-only leaf domain (e.g., Divisions or Stipulations) to prove the pattern, then expand. (6) Explicit non-goals and constraints (e.g., keep per-handler perf, don't bolt on an ORM, preserve existing GSI-backed query shapes). (7) Testability win — in-memory repository for unit tests replaces the current DynamoDB mock drift problem. Deliverable for this todo is the plan doc, not the implementation.
