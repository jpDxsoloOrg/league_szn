# TO-DOS

## Rethink Matches Feature UX - 2026-02-13 08:47

- **Evaluate removing or reworking Matches page** - The Matches page overlaps with Events and has UX issues with its filter options. **Problem:** The "All" filter fetches every match ever created (no pagination/limit), which is wasteful and not useful to users. The "Completed" filter similarly returns all historical completed matches with no recency limit. The Events feature (`/events` with EventDetail, EventResults) already shows matches grouped by event with richer context, making the standalone Matches page potentially redundant. **Files:** `src/components/Matches.tsx:12,22,163-182` (filter state and buttons), `src/components/Sidebar.tsx:93-94` (nav link), `src/components/events/EventDetail.tsx`, `src/components/events/EventResults.tsx`. **Solution:** Options to consider: (1) Remove the Matches page entirely and direct users to Events for match browsing, (2) Keep Matches but remove the "All" filter and limit "Completed" to the most recent N matches (e.g., last 20), (3) Restyle Matches as a "Recent Results" page showing only the latest completed matches. Decide which approach best fits the UX before implementing.

## Add Division Filter to Championships - 2026-02-13 08:50

- **Add division filter to Championships page** - The Championships page displays all belts in a flat grid with no way to filter by division. **Problem:** Users cannot filter championships by division, making it harder to find relevant belts when multiple divisions exist. The `Championship` type already has a `divisionId` field (`types/index.ts:39`) but the component never uses it — it just renders all championships in a single unfiltered grid (`Championships.tsx:129-169`). **Files:** `src/components/Championships.tsx:11,125-169` (state and grid rendering — no division filter exists), `src/types/index.ts:34-43` (Championship type with `divisionId` field). **Solution:** Add a division filter dropdown/button group above the championships grid. Fetch divisions list (or derive unique divisions from the championships data), default to showing all, and filter the `championships` array before mapping. May also want to group championships visually by division.

## Merge Stats Into Profile Page - 2026-02-13 08:52

- **Merge PlayerStats into WrestlerProfile page** - Combine the separate stats and profile pages into a single unified profile view. **Problem:** The Profile page (`/profile`) shows basic info (image, name, wrestler name, W-L-D record) and the Stats page (`/stats`) shows detailed statistics with a player dropdown — but these are separate pages requiring users to navigate between them. The profile page should be the single destination for a wrestler's full view: profile header + detailed stats pre-selected to the logged-in user. **Files:** `src/components/profile/WrestlerProfile.tsx:252-410` (profile header, basic stats, edit form), `src/components/statistics/PlayerStats.tsx:1-36` (player dropdown, stats loading — default to first player instead of current user), `src/App.tsx:80-83` (`/profile` route), `src/components/Sidebar.tsx:113-114,147-148` (separate nav links for Profile and Statistics). **Solution:** Embed PlayerStats (or its data/rendering) into WrestlerProfile below the profile header, with the dropdown defaulting to the logged-in user's playerId. Remove the need to navigate to `/stats` separately for your own stats. Keep `/stats` available as a public route for viewing any player's stats. Update sidebar navigation accordingly.

## Rework Challenges UX and Default View - 2026-02-13 08:56

- **Default challenges to My Challenges and hide Challenge Board** - The challenges nav currently lands on the Challenge Board (`/challenges` -> `ChallengeBoard`), but the more useful default for a wrestler is their own challenges. **Problem:** Users land on the public Challenge Board which may not be the most relevant view. The "My Challenges" view (`/challenges/my`) is more personal and actionable but requires extra navigation. Additionally, consider either integrating challenges into the merged profile/stats page (from the previous todo) or keeping challenges separate but defaulting to My Challenges. The Challenge Board should be hidden/disabled for now. **Files:** `src/App.tsx:87-113` (challenge routes — `/challenges` maps to `ChallengeBoard`, `/challenges/my` maps to `MyChallenges`), `src/components/Sidebar.tsx:116-118,132-134` (nav links point to `/challenges`), `src/components/challenges/ChallengeBoard.tsx`, `src/components/challenges/MyChallenges.tsx`. **Solution:** Either (A) change `/challenges` route to render `MyChallenges` instead of `ChallengeBoard` and hide/remove the ChallengeBoard nav entry, or (B) integrate a My Challenges section into the merged profile/stats page and simplify the challenges routing. For now, option A is simpler — redirect `/challenges` to `MyChallenges` and disable ChallengeBoard.

## Link Promo Call-Outs to Challenges Flow - 2026-02-13 09:00

- **Auto-create challenge when posting a call-out promo and hide Issue Challenge button** - The promo editor already has a `call-out` promo type (`PromoEditor.tsx:19`) that targets another wrestler, but it doesn't create an actual challenge record. The separate Issue Challenge page (`/challenges/issue`) duplicates this intent. **Problem:** Two disconnected paths to challenge someone — the `call-out` promo type and the `IssueChallenge` component — with no integration between them. A call-out promo should automatically create a challenge, and the standalone Issue Challenge button/page should be hidden since promos are the canonical way to issue challenges. **Files:** `src/components/promos/PromoEditor.tsx:19-23,67,118,186` (call-out promo type, form state, submission), `src/components/challenges/IssueChallenge.tsx:11` (standalone issue page to hide), `src/App.tsx:95-100` (`/challenges/issue` route to remove/hide), `src/components/challenges/MyChallenges.tsx:55-56,136-139` (accept action handler and button). **Solution:** (1) When a `call-out` promo is submitted, also call `challengesApi.create()` to create a challenge record targeting the same wrestler. (2) Hide the Issue Challenge button/route. (3) When a wrestler clicks "Accept" on a received challenge in MyChallenges, navigate to `/promos` with query params or state pre-setting promoType to `response` and targetPlayerId to the challenger's ID, so they can cut a response promo.
